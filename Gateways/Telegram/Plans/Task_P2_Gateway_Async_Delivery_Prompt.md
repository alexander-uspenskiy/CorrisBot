# Задача для агента: исправить пункт 2 (ожидание/паника Talker) через асинхронную модель доставки ответов

## Контекст проекта
- Репозиторий: `C:\CorrisBot`
- Платформа: мульти-агентные loopers (файловые prompt/result), пользовательский канал сейчас через Telegram Gateway.
- Ключевые компоненты:
  - `Gateways/Telegram/tg_codex_gateway.py` — gateway Telegram -> Talker looper.
  - `Looper/codex_prompt_fileloop.py` — файловый loop runner (timestamp prompts).
  - `Talker` — основной looper для общения с пользователем.
  - `Orchestrator` и `Workers` — проектные loopers.

## Что наблюдалось (проблема пункта 2)
В реальном тесте Talker после постановки задачи Orchestrator не завершил turn, а начал внутреннее синхронное ожидание с таймаутами/опросом и сообщениями "жду/повторю".  
Из-за этого:
- Talker выглядел как "паникующий" и "самопинающийся".
- Новые сообщения пользователя (например "не повторяй") обрабатывались только после завершения долгого run.

## Подтвержденная причина
`Gateway` сейчас работает в режиме request/response на **конкретный** prompt/result marker:
- На входящее сообщение gateway создает конкретный `Prompt_...md`.
- Затем ожидает строго связанный `Prompt_..._Result.md` для этого marker.
- Пока идет это ожидание, run держит lock и не является "свободным асинхронным роутером ответов".

Это плохо согласуется с архитектурой loopers, где агенты должны обмениваться файлами асинхронно, а ответы могут приходить позже и не обязаны быть частью того же user turn.

## Что уже согласовано и считается целевым поведением
1. Для handoff Talker -> Orchestrator нужен режим "отправил и завершил turn" (fire-and-forget по умолчанию).
2. Ответы должны доставляться пользователю через файловый канал, когда они реально появляются (без привязки к одному ожидаемому marker конкретного user-run).
3. Не нужен polling внутри ответа модели "дождаться оркестратора прямо сейчас".

## Важные недавние правила (учесть, не ломать)
- Добавлен `Reply-To` контракт между Talker и Orchestrator:
  - `Talker/ROLE_TALKER.md`
  - `ProjectFolder_Template/Orchestrator/ROLE_ORCHESTRATOR.md`
- В шаблоне `Talker/AGENTS_TEMPLATE.md` блок `## CRITICAL` должен оставаться, но в собранном `Talker/AGENTS.md` удаляться сборщиком (уже поддержано в `Looper/assemble_agents.py`).

## Цель этой задачи
Переделать `Gateways/Telegram/tg_codex_gateway.py` с модели "ждать конкретный result файла для текущего запроса" на модель:
- `submit` (принять пользовательский prompt и положить в inbox Talker),
- `deliver` (фоновой доставкой стримить новые результаты из inbox Talker в Telegram).

Итог: gateway становится асинхронным мостом, а не блокирующим RPC-ожидателем одного ответа.

## Ограничения и границы
- Изменения в рамках этого пункта — в первую очередь `Gateways/Telegram/tg_codex_gateway.py`.
- Не ломать текущую безопасность (`ALLOWED_CHAT_ID`, scope reset, sanitize sender ids/paths).
- Не возвращаться к индексным prompt-файлам: только timestamp (`Prompt_YYYY_MM_DD_HH_MM_SS_mmm...`).
- Не удалять поддержку текущих команд `/reset*`, `/help`, `/show_*`, `/setconsole` и т.д.
- Если нужны небольшие правки в соседних файлах для корректной интеграции — делай минимально и явно обоснуй.

## Что нужно реализовать (ожидаемый дизайн)
1. Разделить жизненный цикл:
   - При входящем тексте/файле: только enqueue prompt для Talker (быстрый ACK пользователю).
   - Отдельный async background worker/pump: мониторит новые result-события и отправляет в Telegram.
2. Убрать зависимость доставки от единственного `result_path` текущего request-а.
3. Дедупликация:
   - Не слать одни и те же `agent_message`/`DELIVER_FILE` повторно после рестартов.
   - Вести watermark/state (по sender/file/offset/marker — на твой выбор).
4. Порядок:
   - Доставлять события в стабильном порядке (по времени/marker).
5. Управление конкурентностью:
   - Избежать блокировки приема новых сообщений на длительный run.
   - Не допустить гонок между submit и deliver worker.
6. Логирование:
   - Обновить `run.log`/операционные логи так, чтобы было видно submit/deliver этапы и ошибки.

## Что считается успешным результатом (acceptance criteria)
1. Пользователь отправляет задачу на Orchestrator через Talker.
2. Talker быстро завершает user turn (без внутреннего "жду-повторю" цикла).
3. Пользователь сразу после этого может отправить "не повторяй" и сообщение обрабатывается без ожидания завершения старого run.
4. Когда Orchestrator позже кладет ответ в `Reply-To` inbox Talker, gateway самостоятельно доставляет этот ответ в Telegram.
5. Нет дублей сообщений/файлов при нормальной работе и после перезапуска gateway.
6. Нет регрессии по загрузке/доставке файлов (`DELIVER_FILE`).

## Минимальные тестовые сценарии, которые нужно прогнать
1. Базовый async handoff:
   - user -> Talker: "передай оркестратору ...";
   - Talker завершает turn быстро;
   - Orchestrator отвечает позже;
   - gateway доставляет ответ без нового user prompt.
2. Interruption:
   - сразу после handoff отправить "не повторяй";
   - сообщение должно обработаться без timeout/busy-залипания.
3. Restart resilience:
   - перезапуск gateway между submit и поздним ответом;
   - после старта поздний ответ должен доставиться 1 раз.
4. File delivery:
   - ответ с несколькими `DELIVER_FILE`;
   - все файлы отправлены, без повторной отправки тех же файлов.

## Важно
- Не делай "частичное решение" в стиле прежнего wait-for-one-result.
- Не зашивай в поведение Talker внутренние таймаутные циклы ожидания Orchestrator-ответа.
- Сохраняй совместимость с текущими timestamp marker и форматом result-потока.
