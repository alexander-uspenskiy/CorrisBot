# Задача для агента: закрыть пункт 7 (убрать таймаутное ожидание в модели) через `async by default`

## Контекст
- Репозиторий: `C:\CorrisBot`
- Архитектура: file-based loopers (`Prompt_...md` / `Prompt_..._Result.md`) + Telegram Gateway.
- Пункт 2 уже реализован отдельной задачей (см. отчет):  
  `Gateways/Telegram/Plans/Task_P2_Gateway_Async_Delivery_Report.md`
- После п.2 Gateway работает асинхронно (`submit` + background `deliver`), поэтому нет необходимости держать один user turn до "конкретного ответа".

## Проблема пункта 7
В базовых правилах сейчас есть дефолт:
- `Looper/ROLE_LOOPER_BASE.md`:  
  `If a prompt asks to pass work to another looper and report back here, treat it as synchronous by default.`

Из-за этого модель склонна в рамках одного turn:
- запускать внутренние ожидания (`while/sleep/poll`);
- ставить свои дедлайны/таймауты;
- писать "продолжаю ждать", "повторю запрос", и т.п.

Для нашей платформы это лишнее: loopers должны работать на входящих файлах асинхронно, без блокирующего "дожидания здесь и сейчас", если пользователь явно не просил sync.

## Цель
Сделать поведение loopers по умолчанию:
- `async by default`;
- `sync only if explicitly requested`.

Важно: это **не** про параллельную обработку нескольких prompt одновременно.  
Один looper по-прежнему может обрабатывать prompt последовательно (это нормально).

## Что нужно изменить
1. Базовое правило:
   - Обновить `Looper/ROLE_LOOPER_BASE.md`:
     - убрать `synchronous by default`;
     - явно зафиксировать `asynchronous by default`;
     - синхронный режим только при явном запросе пользователя/вышестоящего агента.
2. Специализация Talker:
   - `Talker/ROLE_TALKER.md` привести в согласованное состояние с новым базовым правилом.
   - Убрать/переформулировать пункты, где сейчас зашито "по умолчанию дождись отчета оркестратора".
   - Оставить возможность sync-режима по явной фразе пользователя:
     - например: "дождись ответа и верни в этом же сообщении".
3. Непротиворечивость прочих ролей:
   - Проверить, что `ProjectFolder_Template/Orchestrator/ROLE_ORCHESTRATOR.md` и `ROLE_WORKER.md` не содержат конфликтующих обязательных sync-дефолтов.
4. Сборка AGENTS:
   - Пересобрать `Talker/AGENTS.md` после правок шаблонов/ролей.
   - Убедиться, что шаблонный `## CRITICAL` в `Talker/AGENTS_TEMPLATE.md` остается в шаблоне, но отсутствует в собранном `Talker/AGENTS.md` (текущая сборка должна это уметь).

## Границы задачи
- Не менять код gateway в рамках этого пункта (п.2 уже сделан).
- Не менять модель последовательной обработки prompt внутри одного looper-процесса.
- Не внедрять принудительное прерывание уже выполняющегося prompt.

## Acceptance criteria
1. В `Looper/ROLE_LOOPER_BASE.md` явно закреплен `async by default`.
2. В `Talker/ROLE_TALKER.md` нет требований "по умолчанию ждать отчет в том же turn".
3. В правилах есть явный sync-override только по явному запросу.
4. `Talker/AGENTS.md` пересобран и отражает новые правила.
5. Нет внутренних противоречий между базовой ролью и Talker/Orchestrator ролями.

## Регрессионные проверки (минимум)
1. Статическая проверка:
   - `rg` по ключам `synchronous by default`, `дождись отчета`, `ждем отчет`, `по умолчанию это синхронный`.
   - Убедиться, что остались только корректные формулировки (например, sync "по явному запросу").
2. Поведенческая sanity:
   - Формулировка handoff Talker -> Orchestrator должна завершать turn без обязательного ожидания ответа.
3. Проверка сборки:
   - `Talker/AGENTS.md` без шаблонного `## CRITICAL`.

## Отчет от агента (обязательный формат)
1. Что изменено по файлам (список + суть).
2. Какие формулировки заменены (старое -> новое).
3. Результаты регрессионного поиска/проверок.
4. Остаточные риски (если есть).

## Важно
- Вносить только нужные изменения без "попутного рефакторинга".
- Если встречается неоднозначность формулировок, приоритет:  
  `async by default` + `sync only on explicit request`.
