# План перехода на Windows Terminal

## Цель
Перевести запуск луперов с отдельных всплывающих `cmd`-окон на управляемый запуск в Windows Terminal (WT), сохранив текущую логику `codex_prompt_fileloop.py` без функциональных изменений.

## Что не меняем
- Бизнес-логику лупера в `codex_prompt_fileloop.py`.
- Формат входящих/исходящих prompt-файлов.
- Текущий low-level запуск одного агента через `CodexLoop.bat` (оставляем как legacy-совместимый слой).

## Целевая архитектура запуска
1. `codex_prompt_fileloop.py`:
   выполняет loop и не знает, из какого терминала его запустили.
2. `CodexLoop.bat`:
   low-level вход для запуска одного агента.
3. `StartLoopsInWT.*` (новый launcher):
   high-level запуск нескольких агентов в WT (tabs/panes), с anti-duplicate логикой.

## Модель окон (микс tabs + panes)
- Допускаем несколько WT окон.
- В каждом WT окне:
  - 1-2 рабочих tab для одновременного мониторинга (внутри pane-сетка).
  - Остальные агенты размещаются в дополнительных tabs.
- Рекомендованный стартовый профиль раскладки:
  - Tab `Core`: 4 panes (Orchestrator, Talker, Executor_001, Executor_002).
  - Tabs `Exec-A`, `Exec-B`, ...: по 2-4 агента в pane-раскладке.

## Именование tab/pane
Формат заголовка:
`[<ProjectTag>] <AgentPath>`

Пример:
- `[ProjectFolder_Template] Orchestrator`
- `[ProjectFolder_Template] Executors/Executor_001`

`ProjectTag` берется из имени каталога проекта (родитель `.CorrisBot`).

## Стратегия запуска (fire-and-forget)
Для запуска дочерних луперов использовать `wt`-команды (new-tab/split-pane), без ожидания завершения процесса.

Требование к оркестратору в prompt-инструкции:
- запускать один раз;
- не делать polling/kill/retry в том же шаге;
- считать запуск успешным после успешного старта команды WT.

## Anti-duplicate правило
Перед запуском агента launcher проверяет, есть ли уже активный процесс с тем же сочетанием:
- `codex_prompt_fileloop.py`
- `--project-root <...>`
- `--agent-path <...>`

Если найден:
- новый процесс не стартуем;
- в консоль пишем, что агент уже запущен.

## Конфигурация запуска
Рекомендуется отдельный конфиг-файл, например:
`Plans/loops.wt.json`

Минимальная схема:
- `project_root`
- `windows` (массив):
  - `name`
  - `tabs` (массив):
    - `name`
    - `panes` (массив):
      - `agent_path`
      - `focus` (опционально)

Плюсы конфигурации:
- одинаковый запуск в тестах и проде;
- масштабирование до 20 агентов без правок кода launcher'а;
- база для будущего AgentRunner.

## Совместимость
- Legacy запуск одного агента (`CodexLoop.bat`) остается рабочим.
- Новый WT launcher не ломает текущие `.lnk`.
- Переход можно делать поэтапно: сначала ручной WT launcher, затем интеграция в prompt-flow оркестратора.

## Этапы внедрения
1. Добавить новый WT launcher (`StartLoopsInWT.*`) для статического набора агентов.
2. Добавить anti-duplicate проверку.
3. Добавить конфиг `loops.wt.json`.
4. Перевести существующие `.lnk` на новый launcher.
5. Обновить prompt-шаблоны оркестратора на корректный fire-and-forget запуск через WT.

## Критерии готовности
- Можно поднять Orchestrator + минимум 2 Executors в одном WT окне.
- Не создаются дубли процессов при повторном запуске launcher'а.
- Заголовки tab/pane однозначно показывают проект и агента.
- Legacy `CodexLoop.bat` по-прежнему запускает одиночный лупер.

## Риски
- Различия поведения `wt.exe` между версиями Windows Terminal.
- Ограничения по точному управлению уже открытыми tabs/panes.

## Смягчение рисков
- Сначала поддержать надежный сценарий: "создать/добавить новые tabs".
- Управление тонкой раскладкой panes делать только для нового окна/чистой сессии.
- Для сложных сценариев допустить fallback на второй WT window.
