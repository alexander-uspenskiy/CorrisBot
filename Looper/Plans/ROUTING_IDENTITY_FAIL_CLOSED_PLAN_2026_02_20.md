# ROUTING IDENTITY FAIL-CLOSED PLAN (2026-02-20)

## 1. Goal
Eliminate cross-root missroute in self-improvement/runtime-split scenarios (for example `C:\_RUN_CorrisBot` + `C:\CorrisBot`) without heuristic path guessing.

Primary failure to prevent:
- prompt is written into a structurally valid but wrong root (same folders, different runtime instance), so receiver never consumes it.

## 2. Scope
In scope:
- routing contract for Talker/Orchestrator/Worker prompt transport;
- root identity model (`APP_ROOT`, `AGENTS_ROOT`, `EDIT_ROOT`, `ROUTE_SESSION_ID`);
- fail-closed preflight checks before every cross-looper send.

Out of scope:
- model/profile migration logic;
- business logic of project tasks;
- fallback/autorepair by path similarity.

## 3. Canonical Roots and Identity
Use exactly three explicit roots per project session:

1. `APP_ROOT`
- runtime root where platform stack is launched.
- `Talker` location is deterministic: `TALKER_ROOT = APP_ROOT\\Talker` (derived, not guessed).

2. `AGENTS_ROOT`
- runtime workspace where Orchestrator/Workers of this session live.
- expected structure: `AGENTS_ROOT\\Orchestrator`, `AGENTS_ROOT\\Workers\\...`.

3. `EDIT_ROOT`
- code/repository root being changed by agents.

Session identity:
- `ROUTE_SESSION_ID` (required, opaque unique id, generated by Talker at project-session start).
- `PROJECT_TAG` (required, stable within session).
- `ORCHESTRATOR_SENDER_ID = Orc_<PROJECT_TAG>` (required, stable within session).

Hard rule:
- if any of the fields above is absent/empty/mismatched, transport is blocked (fail-closed).

## 4. Routing Contract (No Heuristics)
First Talker -> Orchestrator message for a session MUST include a machine-readable block `Routing-Contract`:

```text
Routing-Contract:
- Version: 1
- RouteSessionID: <opaque-uuid-like-id>
- AppRoot: <absolute path>
- AgentsRoot: <absolute path>
- EditRoot: <absolute path>
- ProjectTag: <stable tag>
- OrchestratorSenderID: <Orc_ProjectTag>
- CreatedAtUTC: <ISO-8601>
```

Acceptance rules:
- absolute paths only;
- no placeholders;
- no relative paths;
- `TALKER_ROOT` is computed only as `AppRoot\\Talker`;
- no auto-rewrite to "similar" roots.

Persistence:
- Orchestrator stores accepted contract to
  `AGENTS_ROOT\\Orchestrator\\Temp\\routing_contract.json`.
- Contract is immutable for current `ROUTE_SESSION_ID`.
- Contract update is allowed only via explicit new `Routing-Contract` block from Talker.

Mandatory propagation:
- Every inter-looper prompt in the same session MUST include `Route-Meta` block:

```text
Route-Meta:
- RouteSessionID: <same as Routing-Contract>
- ProjectTag: <same as Routing-Contract>
```

- Missing/changed `RouteSessionID` in `Route-Meta` => block transport.
- Parsing rule: `Route-Meta` is recognized only as top-level block (not inside code fence, not quote, not placeholder values).

## 5. Channel Matrix (Deterministic)

1. Talker -> Orchestrator
- Target inbox (deterministic):
  `AGENTS_ROOT\\Orchestrator\\Prompts\\Inbox\\Talker`
- No alternative destinations.

2. Orchestrator -> Worker
- Target inbox (deterministic):
  `AGENTS_ROOT\\Workers\\<WorkerId>\\Prompts\\Inbox\\<OrchestratorSenderID>`
- `<WorkerId>` must exist under `AGENTS_ROOT\\Workers`.
- No routing to `ProjectFolder_Template` or any root outside `AGENTS_ROOT`.
- Transport must be performed only via deterministic helper for this channel (no direct ad-hoc `create_prompt_file.py --inbox ...` in task flow).

3. Worker -> Orchestrator
- Target inbox only from task `Reply-To.InboxPath` sent by Orchestrator.
- That path must be under:
  `AGENTS_ROOT\\Orchestrator\\Prompts\\Inbox\\...`
- Worker reply must carry same `Route-Meta.RouteSessionID`; mismatch => block.

4. Orchestrator -> Talker
- Target inbox only from valid Talker `Reply-To.InboxPath` bound to current `ROUTE_SESSION_ID`.
- Expected base root: `APP_ROOT\\Talker\\Prompts\\Inbox\\...`
- If Reply-To path does not match current session contract, block send.

5. Worker -> Talker (when Talker Skill is explicitly used)
- Default policy: forbidden.
- Allowed only when explicitly enabled by Orchestrator task contract and with explicit `Reply-To` provided in the task prompt.
- Path must match current session contract roots.
- No direct "computed" Talker path by Worker.

## 6. Preflight Gate (Mandatory Before EVERY Send)
Before writing any `Prompt_*.md` into another looper inbox, sender must run transport preflight:

1. Load pinned routing contract for current session.
2. Verify `ROUTE_SESSION_ID` in current task context equals pinned session id.
3. Resolve target path for channel using Channel Matrix rules only.
4. Validate target path is absolute and within required root for this channel.
5. If channel uses `Reply-To`, verify Reply-To is valid and belongs to same session/root policy.
6. Only after checks pass: create prompt via `create_prompt_file.py`.
7. Verify resulting file exists exactly in preflight-approved target inbox.
8. Log evidence (session id, channel, approved target, delivered file path).

Fail-closed behavior:
- any failed check => no send, explicit error log, escalate upstream.
- no fallback, no retry with altered path, no "closest matching" root.

## 7. Precedence Rules (Explicit)
Routing precedence for a sender turn:

1. Valid `Reply-To` in current incoming prompt (if present for this channel) AND it passes session/root checks.
2. Pinned session `routing_contract.json` fields.
3. Deterministic channel path formula from Channel Matrix.
4. Otherwise: hard stop (`routing_contract_missing_or_invalid`).

Forbidden precedence:
- current working directory;
- path examples from docs;
- template roots;
- previous session data with different `ROUTE_SESSION_ID`.

## 8. Logging and Audit
For every cross-looper send, persist one structured audit record (jsonl):
- `timestamp_utc`
- `route_session_id`
- `channel` (`talker_to_orc`, `orc_to_worker`, `worker_to_orc`, `orc_to_talker`, `worker_to_talker`)
- `sender_id`
- `target_inbox`
- `delivery_file`
- `result` (`ok|blocked`)
- `error_code` (if blocked)

Recommended location:
- `AGENTS_ROOT\\Orchestrator\\Temp\\routing_audit.jsonl`
- for Talker-originated handoff: `APP_ROOT\\Talker\\Temp\\routing_audit.jsonl`

## 9. Migration Plan (Phased)

### Phase A: Contract Formalization
- Update role docs (`Talker`, `Orchestrator`, `Worker`, `Looper Base`) with this contract.
- Add strict examples for first-session handoff and subsequent messages.
- Define standard error codes:
  - `routing_contract_missing`
  - `routing_session_mismatch`
  - `routing_target_out_of_scope`
  - `routing_reply_to_invalid`
  - `routing_reply_to_session_mismatch`

### Phase B: Transport Gate Integration
- Integrate preflight checks into existing deterministic helpers (`send_orchestrator_handoff.py`, `send_reply_to_report.py`) without broad redesign.
- Add/enable deterministic helper for `Orchestrator -> Worker` channel with mandatory preflight and delivery verification.
- Ensure all cross-looper writes go through helpers; ban ad-hoc direct `--inbox` usage in role contracts.

### Phase C: Session Persistence
- Implement/write pinned `routing_contract.json` in Orchestrator session root.
- Bind all outgoing sends to this file until explicit contract rotation from Talker.

### Phase D: Test Gates (Must Pass)
Add/extend tests for:
1. happy-path for all channels;
2. two identical roots with different `ROUTE_SESSION_ID` (must block mismatch);
3. target path in wrong root with valid structure (must block);
4. missing `Routing-Contract` on first session message (must block);
5. stale Reply-To from previous session id (must block);
6. missing `Route-Meta` in downstream prompt/reply (must block);
7. ad-hoc direct inbox send bypass attempt (must block by contract/test harness).

### Phase E: Rollout
- Enable contract in strict mode for new sessions.
- Keep no compatibility fallback for ambiguous legacy routing (fail-closed by design).

## 10. Regression Risk Analysis
Main trade-off:
- stricter routing can block delivery earlier.

Why acceptable:
- blocking explicit mismatch is safer than silent wrong-root delivery.
- this project is in active development; deterministic behavior is preferred over permissive compatibility.

## 11. Acceptance Criteria
Plan is considered implemented successfully only if all are true:
1. No prompt can be delivered to a root outside channel scope for current session.
2. Switching between two parallel CorrisBot roots cannot cross-route messages.
3. Every send has audit evidence with `route_session_id` and target path.
4. No routing decision depends on heuristics or inferred "similar" paths.

## 12. Operator Checklist (Runtime)
Before starting orchestration session:
1. Confirm `APP_ROOT`, `AGENTS_ROOT`, `EDIT_ROOT` explicitly.
2. Confirm first handoff contains `Routing-Contract` with new `ROUTE_SESSION_ID`.
3. Confirm Orchestrator persisted `routing_contract.json`.
4. Confirm first delegated Worker task uses expected `AGENTS_ROOT\\Workers\\...` inbox.
5. Confirm first Orchestrator report to Talker uses Reply-To bound to same session id.
