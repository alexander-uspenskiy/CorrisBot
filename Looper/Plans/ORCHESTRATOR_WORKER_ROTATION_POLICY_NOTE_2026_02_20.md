# Orchestrator Note: Worker Rotation Policy (2026-02-20)

## Почему Worker_002 не был создан после PHASE 4

Это было управленческое решение с моей стороны, и ретроспективно оно не оптимальное.

Факторы, из-за которых я продолжил с `Worker_001`:
1. По отчетам на PHASE 2-4 оставался запас контекста (`~45%`, затем `~30%`), и я ошибочно трактовал это как достаточный ресурс для продолжения без ротации.
2. Я переоценил выгоду continuity (меньше handoff-накладных) и недооценил деградацию качества/скорости на длинной серии фаз.
3. В действующей формулировке правил не был жестко закреплен пороговый триггер "создай нового Worker", поэтому я применил слишком консервативную стратегию reuse.

Итог: нового Worker стоило заводить раньше, даже без параллельной работы.

## Как переформулировать правила, чтобы легко создавать новых Workers

Ниже предлагаю норму для инструкций Оркестратора.

### 1) Context Budget Gate (обязательный)
1. `Soft threshold`: если текущая загрузка `>=40%` и впереди medium/large задача, по умолчанию создается новый Worker.
2. `Hard threshold`: при `>=60%` запрещено выдавать текущему Worker новый существенный scope; обязательна ротация.
3. Перед каждым новым этапом Оркестратор фиксирует прогноз: `current_load + expected_delta`; если прогноз `>60%`, создается новый Worker заранее.

### 2) Phase Segmentation Rule
1. Один Worker по умолчанию ведет не более 2 последовательных крупных фаз.
2. На следующую крупную фазу создается новый Worker даже при отсутствии параллелизма.

### 3) Reuse Must Be Justified
1. Если Оркестратор НЕ создает нового Worker при `>=40%`, он обязан явно записать причину (1-2 строки) в task/report.
2. Отсутствие такой причины считается нарушением orchestration policy.

### 4) Mandatory Context Reporting
1. Каждый отчет Worker должен включать:
   - `current context load estimate`;
   - `expected load for next phase`;
   - рекомендацию: `reuse` или `rotate`.
2. Оркестратор обязан принимать это как вход в решение о ротации.

### 5) Bias to Cheap Bootstrap
1. В данной платформе bootstrap Worker дешевый (файлы на диске, без постоянного memory cost), поэтому политика должна быть: `rotate early, not late`.
2. Параллельность не требуется: новый Worker допускается и для последовательного пайплайна.

## Краткая формула решения

`>=40% + впереди сложная задача` => создать нового Worker.

`>=60%` => обязательная ротация до постановки следующей существенной задачи.

## 6) Fail-Closed Context Control (дополнение)
1. Перед выдачей каждой новой medium/large задачи Оркестратор проверяет "свежесть" контекст-оценки.
2. Свежесть определяется как: оценка получена не старше предыдущего этапа.
3. Если свежей оценки нет, задача не выдается до получения обновленного контекст-статуса.
4. Если в отчете отсутствуют поля `current_load` / `expected_delta_next` / `decision`, отчет считается неполным и не принимается.
5. Если Worker не рапортует контекст, Оркестратор применяет fail-safe: трактует ситуацию как high-risk и выполняет ротацию на нового Worker.
6. Для `>=60%` reuse запрещен без исключений: только rotate.
