# Исследование “мусора” `EDIT_ROOT` и `RUN_ROOT` в Telegram

## 1. Current Flow (as-is)
- **Канал доставки:** 
  1. `Worker` (генерирует отчет по шаблону Оркестратора) -> 
  2. `Orchestrator` (принимает отчет и пересылает его Talker) -> 
  3. `Talker` (оборачивает сообщение в YAML `type: relay`, `[Orc_<Tag>]: <verbatim>`) -> 
  4. Script `Gateway` ретранслирует этот relay payload в Telegram.
- **Источники внедрения:** 
  - Явного хардкода строк `"3) EDIT_ROOT:"` и `"4) RUN_ROOT:"` в python-скриптах нет.
  - Поля формируются LLM (Orchestrator/Worker) в теле markdown-отчетов (`C:\CorrisBot\Temp\PHASE..._REPORT.md:3-4`).
  - Данные для этих полей LLM берет из `Routing-Contract` (`EditRoot`, `AgentsRoot`), который инжектится скриптом `send_orchestrator_handoff.py:96-103`.
  - По инструкциям `ROLE_ORCHESTRATOR.md:142`, Оркестратор обязан сам выдавать шаблон отчетности для Worker. Оркестратор "изобрел" этот заголовок для полноты контекста (эхо контракта) и тиражирует его от фазы к фазе.

## 2. Root Cause
- **Причина:** LLM (Orchestrator) считает переменные окружения и пути (Edit/Run roots) важной частью контекста операции и отражает их в человекочитаемом тексте отчета как защиту от потери контекста. В `ROLE_TALKER.md:140` закреплен строгий `VERBATIM relay contract`: Talker передает текст отчета пользователю в Telegram "без купюр".
- **Характер проблемы:** Это не технический баг скриптов, а побочный эффект *design choice* (verbatim relay) в сочетании с чрезмерной дисциплинированностью LLM (сохранение routing/path identity в теле отчета). 

## 3. Invariants (must keep)
- **Routing Identity & Portability Split:** Поля `EditRoot` и `AgentsRoot` (Run root) критически важны для машинной маршрутизации (согласно `ROUTING_IDENTITY_FAIL_CLOSED_PLAN_2026_02_20.md:12`). Их удаление из `Routing-Contract` недопустимо.
- **Fail-closed Delivery:** Контракты `Message-Meta` (`report`/`trace`) и `Reply-To` должны продолжить работать без изменений.
- **Verbatim Relay Rule:** Запрещено давать LLM (Talker) свободу "креативно" модифицировать или пересказывать reports от Оркестратора. Транзит должен оставаться предсказуемым.

## 4. Варианты решения

### Вариант A: Инкапсуляция в Message-Meta (Strict Data vs View Separation)
- **Идея:** Запретить LLM выводить пути конфигурации в текстовом теле отчета. Если Оркестратор хочет зафиксировать пути для аудита, он обязан передавать их только в машинно-читаемом блоке (например, в `Message-Meta` или `Routing-Contract`).
- **Слои:** `ROLE_ORCHESTRATOR.md` и `ROLE_WORKER.md` обновляются инструкцией: "Never include operational environment lines (like `EDIT_ROOT`, `RUN_ROOT`, `AGENTS_ROOT`) in the human-readable report body. If audit is required, keep them in `Message-Meta`."
- **Плюсы:** Идеальная архитектурная чистота. Verbatim-контракт Talker не страдает. Никаких эвристик.
- **Риски:** Требует строгого следования инструкциям от LLM. В редких случаях LLM может забыть про запрет.
- **Сложность:** Low (только правки инструкций ролей).
- **Совместимость:** 100%.

### Вариант B: Разделение Payload на Summary и Tech Details
- **Идея:** Обязать LLM Оркестратора/Worker разбивать отчет на `USER SUMMARY` (только суть, без путей) и `TECH AUDIT` (пути, токены, контексты). Talker или Gateway извлекает и отправляет в TG только секцию `USER SUMMARY`.
- **Слои:** Инструкции агентов (форматирование отчетов) и `ROLE_TALKER.md` (или python-мост), вырезающий тех-часть через регулярное выражение или разделитель.
- **Плюсы:** Скрывает и другой потенциальный технический мусор.
- **Риски:** LLM может ошибиться и поместить важный вопрос пользователю в Tech секцию (срезав его). Это эвристика парсинга текста. Нарушает чистый Verbatim Contract.
- **Сложность:** Medium.
- **Совместимость:** Ломает ожидания по бесшовному Verbatim-роутингу. Риск регрессии (Message/Report loss).

### Вариант C: Фильтрация-эвристика на уровне Gateway
- **Идея:** На уровне python-скрипта моста Telegram (например, `send_to_telegram.py`) добавить regex-фильтр, удаляющий любые строки, содержащие `EDIT_ROOT:` или `RUN_ROOT:`.
- **Слои:** Gateway Layer.
- **Плюсы:** Не зависит от послушности LLM. Гарантированно очищает канал.
- **Риски:** Типичная хрупкая эвристика. Если пользователь намеренно спросит "какой у нас EDIT_ROOT?", ответ бота будет некорректно отфильтрован по регулярке.
- **Сложность:** Low/Medium (написание надежного regex).
- **Совместимость:** Частично нарушает fail-closed прозрачность канала. Запрещено в контексте "Избегать эвристики".

### Вариант D: Структурированный блок User-Payload (Детерминированный подход)
- **Идея:** Отказаться от сплошного парсинга всего тела отчета как пользовательского сообщения. Ввести обязательный блок (например, `User-Message:` или `---BEGIN USER OVERVIEW---`), внутри которого формируется текст *только* для пользователя. Talker будет извлекать из отчета Оркестратора строго этот блок и ретранслировать его, игнорируя остальной "технический" Markdown. 
- **Слои:** `ROLE_ORCHESTRATOR.md` (научить формировать блок) + `ROLE_TALKER.md` / `Looper scripts` (научить детерминированно извлекать блок для релея вместо отправки всего тела).
- **Плюсы:** Полностью решает проблему prompt bleed. Свободно от эвристики. 
- **Риски:** **Критический архитектурный изъян.** Нарушает принцип `VERBATIM` релея для Talker и вводит хрупкость (если LLM забудет блок — сообщение потеряется, нарушив fail-closed).
- **Сложность:** Medium/High 
- **Совместимость:** Запрещено из-за риска silent drops (потери сообщений).

### Вариант E: Изоляция машинных заголовков (Machine-Header Isolation)
- **Идея:** Устранить саму первопричину — видимость транспортных путей для LLM. Вместо того чтобы фильтровать текст *на выходе*, нужно не показывать машинно-маршрутные заголовки (`Routing-Contract`) *на входе*. Скрипт-раннер (напр. `codex_prompt_fileloop.py`) должен парсить заголовки `Route-Meta` / `Routing-Contract` из Inbox-файла для нужд роутинга, но в промпт для LLM передавать **только Payload** (чистое сообщение/инструкцию).
- **Слои:** Уровень Agent Runner (`codex_prompt_fileloop.py` или аналогичный компилятор промптов).
- **Плюсы:** 100% гарантия отсутствия путей в ответе. Идеальная архитектурная чистота (строгое разделение Transport Layer и Logic Layer). Talker остается прозрачной трубой (`VERBATIM` не нарушен, fail-closed не страдает). 
- **Риски:** Машинные заголовки больше не будут доступны LLM для "понимания контекста". (Но они LLM и не нужны — LLM должна думать только над задачей, а не над технической маршрутизацией).
- **Сложность:** Medium (рефакторинг функции извлечения промпта перед подачей в модель).
- **Совместимость:** 100% совместимость с транспортной архитектурой проекта.

## 5. Рекомендуемый вариант
**Вариант E (Изоляция машинных заголовков)**.
- **Почему лучше:** После детального (Deep) Code Review выявилось, что все предыдущие варианты (A, B, C, D) лечат симптомы на стороне транспортного слоя (Gateway/Talker), рискуя сломать строгие контракты доставки. **Option E** лечит саму причину: прячет системную конфигурационную "внутрянку" от глаз языковой модели, оставляя транспортный слой абсолютно надежным и нетронутым.
- **Безопасность:** Исключает любую возможность потери пользовательских сообщений (silent drops), так как Talker продолжит ретранслировать всё verbatim.
